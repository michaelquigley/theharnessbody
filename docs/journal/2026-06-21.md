# 2026-06-21

Bootstrapped the project and built the reviewer subsystem. Notes a future agent
couldn't recover from the tree:

## The de-risk result (live state)

The codex/schema 400 that stalled otis was the first thing to settle. Root cause
was misdiagnosed as JSON-Schema strictness; it's actually structural — nested
objects/arrays *inside array items*. Full reasoning is in
`docs/current/reviewer-output-contract.md` and the `GuardEnvelope` doc comment.

Confirmed empirically (`reviewer/confirm`, `-tags integration`): **codex**,
**claude**, and **pi** all accept and return valid output for the findings schema.
The de-risk is fully closed across the three backends. (codex and claude were
confirmed first; pi once it was logged into gpt.) The harness skips a backend that
is present-but-unauthenticated (see `isAuthError`), so the suite stays green where
a backend isn't usable.

## Deliberate non-actions (don't "fix" these)

- **Dropped mercurius's `SessionMeta` (SessionID/RoundNumber)** from
  `reviewer.ReviewRequest`. It's broker-specific and unused by any adapter; the
  body's reviewer interface stays tool-agnostic. Callers put round context into
  the prompt, which is already how it works.
- **Removed `ReviewRequest.Artifacts` / the `Artifact` type too.** Lifted from
  mercurius but inert here — no adapter reads it (mercurius's *prompt assembly*
  consumes artifacts, inlining them into `Prompt`; the reviewer only ever sees
  `Prompt`). The interface is now exactly what reviewers consume: `Prompt`,
  `Schema`, `WorkingDir`. Don't re-add it from mercurius during the refactor; if a
  `prompt/` subsystem ever needs an artifact abstraction, define it there, where
  it's actually used.
- **The body does not carry mercurius's domain validators** (`ValidateUniqueIDs`,
  `ValidateReviewConsistency`, the `ReviewOutput`/`Concern` types). Those are tool
  concerns; the body validates against a *supplied* schema and owns no canonical
  output types. Keep it schema-agnostic.
- **`GuardEnvelope` is scoped to array-item nesting only** — the construct we have
  evidence for. It does not flag root-level nested objects; don't broaden it
  without a measured reason.
- **codex adapter diverges from mercurius on purpose**: `config.toml` is optional
  (link-if-present) and there's a per-reviewer `Env` merge. Both are otis lessons
  (codex runs on defaults; a service env may lack PATH).

## Couplings / decisions

- The otis nested-`location` / `bok_refs` schemas in
  `reviewer/schema/schema_test.go` are the regression anchor for the guard — they
  encode the lesson. If the guard changes, they're the test that should still bite.
- `reviewer-output-contract.md` lives in `docs/current/` (not `docs/future/` as
  the plan's wording said) because it describes built reality, per the
  current/future split.
- Roadmap order (`docs/future/roadmap.md`) deliberately diverges from
  `harvest-map.md`: `terminus` is the first consumer; the mercurius/sexton refactor
  is deferred to last, not used as an early forcing-function.
- `bok_refs` is intentionally absent from the findings schema — BoK stays
  tool-side (the `otis-bok` repo is the tell). This is the live edge of the open
  "is BoK core/flavor/tool" decision in the roadmap.
