# 2026-06-22

Review-driven fixes to `reviewer/`. Durable decisions a future agent shouldn't have to re-derive:

## GuardEnvelope rejects $ref (resolved)

The `$defs` walk gave the appearance of `$ref` support without resolving references, so an array item hidden behind `$ref` (`{"$ref": "#/$defs/Finding"}`) bypassed the nested object/array guard — worse than no guard, because it's false confidence. Resolved by rejecting any `$ref` outright (codex structured output is unproven with `$ref` anyway; mercurius's schemas are fully inline). Chose reject-over-resolve to keep the guard simple. Regression: `TestGuardEnvelopeRejectsRef`.

## Reviewer adapters are fail-closed (resolved)

The adapters accepted parsable output even when the subprocess exited nonzero, which masks a failed run — and for a *review* substrate a masked failure looks like a clean review. Now a nonzero exit, timeout, or cancellation always returns an error in codex/claude/pi, with captured stdout/stderr kept as diagnostic.

This drops mercurius's recovery path (`recovered_last_message_after_error`) deliberately: no body-level evidence it's needed, and fail-closed is the right default for a substrate. **When mercurius refactors onto the body**, decide whether to port the recovery — and if so, with the measured codex case that justifies it, scoped so a cancelled/timed-out context still always wins.

## jsonout keeps prose-embedded extraction (decided)

`jsonout.firstJSONObject` (bare → fenced → first embedded object) stays. A reviewer flagged it as more forgiving than the "one JSON object" contract — and the first object could be the wrong one — but it's the safety net for the schema-unenforced pi path and claude's result-text fallback (codex and claude's `structured_output` are already clean, handled by bare/fenced). Michael's call: keep the forgiveness, accept the small wrong-object risk on free-form output.

## Mattermost + command harvested from sexton

Built `mattermost/` (transport, lifted ~verbatim from sexton's `client.go`) and `command/` (generic registry, generalized from sexton's `Dispatch`). Durable points:

- **df pinned to v1.0.1 (decided).** The body uses `github.com/michaelquigley/df` **v1.0.1** (latest) by choice. sexton/mercurius are still on **v0.3.x**; when they refactor onto the body they upgrade to v1.0.1 (Go modules raise the whole build to it anyway). Direction is set — donors move up to the body's version, not the body down.
- **CommandHandler → Responder.** sexton's `Client` called a hardcoded `CommandHandler` (status/sync/snooze/resume). The body's `Client` takes a `Responder func(ctx, command) string` callback instead, so `mattermost` imports nothing from `command` and the app wires them (`mc.Start(reg.Dispatch)`). Don't re-couple them.
- **No default trigger word.** sexton defaulted `TriggerWords` to `["sexton"]`; the body can't presume a tool name, so empty `TriggerWords` means mention-only.
- **ctx added.** Handlers/Responder take a context cancelled by `Stop()` (sexton had none), so a long command cancels on shutdown. The client's ctx is created only after the WebSocket connects; `handleMessage` falls back to `context.Background()` for direct (test) construction.
- **Dropped (YAGNI):** `MattermostAlerter`/`AlertEvent`, all sexton-domain commands + formatters, and otis's ~12 KB message truncation (add when terminus posts large findings; Mattermost 413s on oversized posts).

## repo harvested from otis + sexton (read + write)

Built `repo/` combining otis's read-only checkout (mirror/fetch/resolve/worktree, path-based free functions) and sexton's working-tree git handle. Michael chose to include the write side now, anticipating sexton's refactor. Durable points:

- **Two shapes, one package.** The mirror/worktree side is package-level free functions taking directory paths (otis's `MirrorDir`/`state.StateRoot` coupling dropped — the caller owns layout). The working-tree side is a `Repo` handle (sexton's `Git` renamed; `repo.Git` reads oddly).
- **Worktrees come off the bare mirror.** `CreateWorktree(gitDir, sha, scratch)` runs `git -C <mirror> worktree add --detach` — a `--mirror` clone supports worktrees, so the ephemeral checkout is cut straight from the cache.
- **SSH unified with accept-new.** otis's `sshEnv` and sexton's `buildSSHCommand` merged into one builder: `ssh -i '<key>' -o IdentitiesOnly=yes -o StrictHostKeyChecking=accept-new`. Kept otis's `accept-new` (an automated run shouldn't hang on a host-key prompt); this changed sexton's exact SSH string, reflected in the test.
- **Dropped (YAGNI):** sexton's `GenerateCommitMessage` (branded `"sexton: …"` heuristic — callers own commit messages) and otis's state-root coupling.
- **Tests:** real-`git` temp repos for the checkout flow (incl. a stale-ref regression for the explicit-refspec fetch) + sexton's fake-`git`-on-`PATH` shim for the write-op argument/error-mapping assertions.

## scope harvested from otis

Built `scope/` (file selection + byte-budgeted content), lifting otis's `scope.go` + `prompt/scope_content.go` into one package. Durable points:

- **One package, two otis sources.** Resolution lived in otis's `dispatcher`, content-building in its `prompt` package; they're one concern, so they're one `scope` package here.
- **Config coupling dropped.** otis took a `*config.ProjectScopeConfig` and a custom `config.ParseDuration`; the body's `Spec` takes plain `Type`/`Paths`/`Window time.Duration` — the tool owns config and how it parses durations (otis allowed `7d`-style windows; the body leaves that to the caller).
- **Own thin git helpers, not repo's.** scope runs read-only git against a worktree path and keeps its own tiny exec wrapper rather than depending on `repo` — both just take paths, so they stay decoupled and compose at the caller (repo checks out, scope selects).
- **Tests:** real-`git` temp repos with dated commits (`GIT_*_DATE`) and a fixed `now` to make recent-window resolution deterministic; budget truncation asserted via the `<orig>-><kept> bytes` manifest notes.

## prompt harvested from otis + mercurius (primitives, not a builder)

Built `prompt/` as a small set of domain-neutral primitives, not a `Build`. otis and mercurius each have a `Build(req) -> (text, schema)` but their middle sections are genuinely different (otis: BoK + scope + prior findings; mercurius: review calibration + verdict defs + settled decisions), so forcing one builder would be a straitjacket. Durable points:

- **Kept in the body (domain-neutral):** `Fence`/`FencedBlock`, `ScopeContent` (renders `scope.Content`), `SchemaBlock`. Left in the tools (domain-specific): BoK rendering, prior-findings continuity, verdict/severity defs, the what-to-flag rubric, settled decisions — consistent with BoK being tool-side.
- **Fence reconciliation.** otis's `fenceFor` and mercurius's `dynamicFence` were near-identical but disagreed on the minimum (otis floored at 4 backticks, mercurius at 3). The body's `Fence` uses min 3 (standard markdown), always one longer than the longest backtick run in the content.
- **`prompt` imports `scope`** (one-directional) for the `Content` type, and takes the schema as a `json.RawMessage` so it doesn't import `reviewer/schema`. So prompt is the connective tissue: it renders what `scope` selected and what `reviewer/schema` defines, and the tool supplies the prose.

## Code review — resolved

A review pass flagged API-boundary issues from the generalization (not donor-domain behavior to preserve). All resolved, with tests:

- **codex CODEX_HOME moved out of the reviewed tree (blocker).** `prepareCodexHome` created `.codex-home-*` — holding a symlink to live `auth.json` — *inside* `WorkingDir`. Harmless for mercurius (a scratch round dir) but a credential leak in the body, where `WorkingDir` is the code under review. Now created in the system temp dir, outside the worktree.
- **scope recent honors the byte budget.** `KindRecent` bypassed `Options` and inlined full diffs; file content and diffs now share one `budgetedInline` path (per-file + total caps, truncation manifest).
- **empty recent → empty content, not an error.** A window with no commits yields empty `Content` instead of a "base sha required" error.
- **path scope stays in tracked repo content.** `resolvePathScope` rejects `..`/absolute entries and constrains explicit files and glob matches to `git ls-files` (untracked glob matches skipped; an explicitly-named untracked file errors).
- **mattermost lifecycle is an explicit one-shot state machine.** `Start(nil)` responder is an error. State (idle → starting → started → stopped) is guarded by a dedicated mutex: `Start` rejects a second call, a failed `Start` reverts to idle (retryable), and `Stop` is a no-op unless currently started. (The first cut used `sync.Once` + a `started` flag; a follow-up review caught that a pre-`Start` `Stop` consumed the `Once` and left a later successfully-started client unstoppable — the state machine fixes that. Regression: `TestStopBeforeStartThenRealStop`.)
- **dropped `Config.ChannelID`** (unused; `PostMessage` takes the channel explicitly) and **`repo.New` now returns `(*Repo, error)`** with `ErrNotARepo` instead of a bare nil.

## Donor-context audit (verbatim lifts)

A targeted pass over the verbatim lifts with one lens: an assumption safe in the donor's environment but not the body's. Main finding:

- **Reviewers must not run inside the reviewed checkout.** codex and claude auto-load a working directory's `AGENTS.md`/`CLAUDE.md` as instructions (confirmed for claude: `-p` loads `CLAUDE.md` up the tree; `--permission-mode plan` doesn't change it; `--bare` disables auto-discovery but is tied to api-key-only auth, so not a free swap). mercurius never hit this — its `WorkingDir` was a scratch round dir. In the body, `WorkingDir` would be the untrusted code under review, making the repo's own agent-config a prompt-injection channel into the reviewer. pi already guards with `--no-context-files`; codex/claude don't. **Resolution:** documented the contract on `reviewer.ReviewRequest.WorkingDir` — it's a neutral run dir, NOT the reviewed checkout; deliver code via `Prompt`. The API already supports this — `scope` takes its own worktree path, decoupled from the reviewer's `WorkingDir`. Per-backend context suppression (claude `--bare`, a codex equivalent) is optional defense-in-depth, deferred to terminus-wiring time.
- Minor: pi's `--no-context-files` comment rationale was mercurius-specific ("above the round directory") — reframed. mattermost `AllowedUsers` empty = anyone in the channel (sexton's personal-use default) — documented on the field.
- Clean: repo, scope (already hardened), jsonout, dummy, command. codex `CODEX_HOME` was the other instance of this same class, fixed the round before.
